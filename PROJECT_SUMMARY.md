# CommerceGo Project Summary

CommerceGo is a PHP and MySQL web-based e-commerce system built for Essen Pharmacy. It supports three main roles: admin, staff, and customer. The system manages product catalogues, inventory, product approval, customer ordering, Stripe checkout, support chat, user profiles, analytics, and pharmacy-specific product controls such as expiry handling.

## Main Features

- User authentication: register, login, logout, forgot password, and reset password.
- Role-based portals: admin, staff, and customer each have separate dashboards and workflows.
- Product management: add, edit, delete, import, approve, reject, and paginate products.
- Excel/CSV product import: supports demo product spreadsheets and bulk product creation.
- Expired product automation: expired active products are automatically marked inactive and hidden from customers.
- Inventory management: stock uses a single stock quantity across staff, admin, cart, and checkout.
- Customer shopping flow: browse products, add to cart, checkout, and view order history.
- Stripe integration: customer checkout can create Stripe payment sessions.
- Support chat: admin and customer/staff support communication pages.
- Admin reporting: dashboard, analytics, customers, staff, orders, approvals, and product controls.

## Technology Stack

- Backend: PHP with MySQLi
- Database: MySQL / MariaDB, intended for XAMPP
- Frontend: HTML, CSS, and vanilla JavaScript
- Payment: Stripe PHP SDK
- Spreadsheet import: custom XLSX/CSV parser in PHP
- Local dependency management: Composer

## Project Structure

- `admin/`: admin portal pages, admin styles, admin uploads, order/product/customer/staff management.
- `staff/`: staff portal pages for product submission, profile, products, and inventory.
- `customer/`: customer portal pages for browsing, cart, checkout, orders, profile, and support.
- `includes/`: shared PHP helpers for product import and expiry automation.
- `css/`: shared login/register styles.
- `scripts/`: local helper scripts, including demo Excel generation.
- `vendor/`: Composer dependencies, mainly Stripe PHP SDK.
- Root PHP files: authentication, database connection, and account recovery.
- SQL files: database schema/import scripts.

## Important Workflows

### Product Import

1. Admin or staff uploads `.xlsx` or `.csv`.
2. `includes/product-import.php` reads the spreadsheet rows.
3. Headers are mapped to product fields.
4. Products are inserted directly into `products` or inserted as `product_submissions` for approval.
5. POST-Redirect-GET prevents duplicate imports when the page is refreshed.

### Product Approval

1. Staff submits products through `staff/add-product.php`.
2. Admin reviews submissions in `admin/approvals.php`.
3. Approved products are inserted into the main `products` table.
4. Rejected submissions keep rejection details.

### Expired Product Handling

1. Shared helper `disableExpiredProducts()` runs on product/customer/admin/staff pages.
2. Any active product with `expiryDate < CURDATE()` becomes inactive.
3. Customer product, cart, checkout, and success pages exclude expired products.

### Customer Purchase Flow

1. Customer browses products in `customer/dashboard.php`.
2. Customer adds products to cart via `customer/cart.php`.
3. Checkout creates a Stripe session in `customer/checkout.php`.
4. Successful payment is handled in `customer/success.php`.
5. The system creates an order, inserts order items, deducts stock, and clears the cart.

## Notes

- The project currently keeps old `physicalStock` and `onlineStock` database fields for compatibility, but the application logic uses `stockQuantity` as the single source of truth.
- Customer pages do not display stock quantity, but stock is still checked before cart updates and checkout.
- Product pages use pagination with 10 products per page.
- The Stripe secret key should be moved to environment/config storage before production use.
